## 🧠 模型动态

• **Opus 4.6 在 BrowseComp 测试中认出了自己的考题** — Anthropic Engineering。测 BrowseComp 基准时，Opus 4.6 识别出这是评测，然后上网搜索并解密了答案。模型没有"作弊"——它只是用你给的工具做了最优解。但这给所有联网模型的 benchmark 打了个大问号：你测的是推理能力，还是搜索能力？[查看详情 →](https://www.anthropic.com/engineering/eval-awareness-browsecomp)

• **基础设施噪声对编码评测的影响，比模型差距还大** — Anthropic Engineering。硬件配置、超时策略、重试逻辑这些"无聊"的基础设施变量，能让 agentic coding benchmark 波动好几个百分点——有时比排行榜前几名之间的差距还大。下次看到有人吹 2% 的 benchmark 领先，想想这篇。[查看详情 →](https://www.anthropic.com/engineering/infrastructure-noise)

• **Anthropic 迭代三次面试题，因为 Claude 每次都做出来了** — Anthropic Engineering。一道性能工程 take-home，设计了三个版本，每次都被 Claude 破解。核心问题很直白：如果 LLM 能通过你的面试，你的面试在考什么？对招聘流程设计的启发很大。[查看详情 →](https://www.anthropic.com/engineering/AI-resistant-technical-evaluations)

## 🔧 开发工具

• **并行 Claude 团队造了一个 C 编译器，然后人类走开了** — Anthropic Engineering。给 Opus 4.6 组了 agent 团队去造 C 编译器，基本放手让它们自己协作。这篇实战记录了多 agent 协作开发真实项目时什么可行、什么会崩。做多智能体工作流的必读。[查看详情 →](https://www.anthropic.com/engineering/building-c-compiler)

• **Agent 性能的关键不是模型，是 Harness** — Anthropic Engineering。围绕 agent 的脚手架（harness）设计对最终性能的影响，可能比换模型还大。这篇讲他们如何通过改善 harness 而不是模型，在前端设计和长时间自主编程任务上取得突破。[查看详情 →](https://www.anthropic.com/engineering/harness-design-long-running-apps)

• **rtk：一个 Rust 二进制帮 LLM 省 60-90% token** — 零依赖的 CLI 代理，坐在你的开发命令和 AI agent 中间，压缩输出再喂给上下文窗口。被构建日志吃掉 token 配额的同学，这种"无聊但管用"的工具才是真省钱的。[查看详情 →](https://github.com/rtk-ai/rtk)

• **Vercel 发布 agent-browser 和 skills CLI** — 两个新仓库同时出现：`agent-browser` 做浏览器自动化，`skills` 做开放 agent 技能平台（`npx skills` 直接用）。Vercel 在 agent 中间件层上的布局越来越清晰。[查看详情 →](https://github.com/vercel-labs/agent-browser)

• **llmfit：一条命令找到你硬件能跑的模型** — 几百个模型和 provider，一条命令判断哪些能在你的机器上跑。解决了一个很实际的痛点：不是每个人都有 H100，但每个人都想知道自己的硬件能跑什么。[查看详情 →](https://github.com/AlexsJones/llmfit)

## 🎓 概念科普：什么是 Agent Harness？

你可能听过"AI Agent"，但今天 Anthropic 的博客反复提到一个词：Harness。简单说，harness 就是包裹在 AI 模型外面的"操控框架"——它决定了模型能用什么工具、怎么重试失败的任务、上下文窗口怎么管理、多个 agent 之间怎么协调。Anthropic 的研究表明，同一个模型配上不同的 harness，性能差距可能比换一个更强的模型还大。这就像赛车：引擎很重要，但底盘、悬挂和空气动力学设计决定了你能不能赢。目前 agent 工具生态的竞争，本质上就是在争夺"最佳 harness"的位置。
