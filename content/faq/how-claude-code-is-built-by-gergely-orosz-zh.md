# How Claude Code is built - by Gergely Orosz

## Gergely Orosz 的深度访谈揭示了什么？

Gergely Orosz 是知名技术博客 The Pragmatic Engineer 的作者，他与 Claude Code 的两位创始工程师进行了深度对话。这篇文章罕见地展示了这款 AI 开发工具的内部构建方式，以及它可能对软件开发未来的影响。

根据访谈内容，Claude Code 团队的工作方式与传统工程团队截然不同。最引人注目的数据是：**Claude Code 自己编写了 90% 的代码**。工程师平均每天完成约 5 个 PR（Pull Request），而在团队规模翻倍的情况下，每位工程师的 PR 产出反而提升了 67%。

Orosz 在推特上表示，这次访谈让他"窥见了未来"，也让他理解了为什么 Anthropic CEO Dario Amodei 在六个月前预测"90% 的代码将由 AI 编写"。Claude Code 团队在原型设计、产品发布上都更加快速，并且做出了更大胆的技术选择——比如用"vibe code"方式实现 Markdown 渲染器。他们在开发中大量使用 AI，几乎贯穿整个流程。

这篇文章对 AI 开发者和产品经理的核心启示是：AI 辅助开发不仅能提升个人效率，还能在团队扩张时保持甚至提高人均产出。这打破了传统软件工程中"人月神话"的规律。

### Claude Code 团队的工作方式有何独特之处？

Claude Code 团队与 Orosz 过去见过的任何工程团队都不同。他们的特点包括：

- **更快的原型迭代**：不追求完美方案，快速验证想法
- **更快的产品发布**：缩短从开发到上线的周期
- **更大胆的技术决策**：例如用非传统方式实现 Markdown 渲染器
- **全流程 AI 辅助**：从代码编写到代码审查，AI 参与每个环节

这种工作模式代表了一种新的工程文化，传统的代码审查流程和开发节奏可能需要重新定义。

### 90% 代码由 AI 编写意味着什么？

这个数据说明 Claude Code 团队已经将 AI 从"辅助工具"转变为"核心生产力"。工程师的角色从"编写代码"转向"指导、审查和整合 AI 生成的代码"。

这并不意味着工程师变得不重要。相反，工程师需要具备更强的架构设计能力、代码审查能力，以及与 AI 高效协作的技能。人均 5 个 PR/天的产出，说明工程师仍在深度参与开发流程，只是工作内容发生了本质变化。

### 团队扩张时效率为何不降反升？

传统软件工程有个经典定律：增加人手往往导致沟通成本上升，人均产出下降。但 Claude Code 团队在规模翻倍后，人均 PR 产出反而提升了 67%。

原因可能在于：AI 承担了大量重复性编码工作，减少了工程师之间的代码依赖和等待时间。每位工程师可以更独立地推进任务，团队协调成本相对降低。

### 这对 AI 开发者和产品经理有什么实际启示？

对于正在评估 AI 开发工具的团队，这篇文章提供了几个关键参考：

1. **重新定义工程师角色**：招聘时可能更看重架构能力和 AI 协作能力，而非纯编码速度
2. **调整开发流程**：传统的代码审查和发布节奏可能需要加速
3. **评估工具 ROI**：人均产出提升 67% 是一个有价值的基准数据
